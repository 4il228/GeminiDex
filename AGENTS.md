# AGENTS.md: ИИ-Контекст и Правила Команды Разработчиков

Данный документ определяет роли, контекст, технические ограничения и правила взаимодействия для автономных агентов (или инженеров), работающих над созданием расширения для Яндекс Браузера с интеграцией Gemini API.

> **ВАЖНО:** Фазы разработки и детали задач описаны в файле `SPEC.md`. Данный документ (`AGENTS.md`) содержит общие правила, роли и ограничения. Перед началом работы агент ОБЯЗАН прочитать `SPEC.md`.

---

## 1. Общий контекст проекта (Глобальный Промпт)
* **Продукт:** Расширение для Яндекс Браузера (Chromium, Manifest V3).
* **Цель:** Добавление ИИ-агента Google Gemini на любую веб-страницу через инжектируемый интерфейс (Sidebar/Widget).
* **Ключевая фича:** Авторизация пользователя и динамическое изменение доступных функций/лимитов контекста ИИ в зависимости от его текущего **тарифного плана** (Free, Pro, Ultra).
* **Стек:** JavaScript (ES6+), Web Extensions API (Manifest V3), HTML5/CSS3 (изолированный Shadow DOM), Node.js (Backend-прокси для валидации тарифов и сокрытия Gemini API Ключей).
* **Спецификация:** Все фазы разработки, acceptance criteria и acceptance tests описаны в [`SPEC.md`](./SPEC.md).

## 2. Матрица Ролей Агентов

### Роль 1: Архитектор (Software Architect)
* **Зона ответственности:** Структура проекта, безопасность API, управление контекстом Manifest V3, схемы данных.
* **Критерий успеха:** Создание масштабируемой структуры без конфликтов между Content Scripts и скриптами страниц, безопасность токенов.

### Роль 2: Фронтенд-Инженер (Frontend Engineer)
* **Зона ответственности:** Инжектируемый UI (UI/UX агента), изоляция стилей через Shadow DOM, обработка пользовательского ввода, стриминг ответов (SSE/Chunks) от Gemini.
* **Критерий успеха:** Интерфейс не ломает верстку родительского сайта, работает плавно, поддерживает Markdown и подсветку кода в ответах ИИ.

### Роль 3: Бэкенд-Инженер (Backend & Security Engineer)
* **Зона ответственности:** Создание прокси-сервера (Node.js/TypeScript), валидация тарифных планов пользователя, маппинг тарифов на лимиты Gemini (размер контекста, выбор модели, RPM/TPM).
* **Критерий успеха:** API-ключ Gemini никогда не передается на клиент (в расширение). Запросы режутся на бэкенде, если превышен лимит тарифа.

## 3. Технические Стандарты и Ограничения (Строгие правила)

### Разработка расширения (Manifest V3):
1.  **Никакого `eval()` или внешних скриптов:** Весь код расширения должен быть локальным (требование безопасности Chrome Web Store / Яндекс Каталога).
2.  **Изоляция UI:** Интерфейс агента на странице ДОЛЖЕН рендериться внутри `Shadow DOM`. Это предотвратит утечку CSS стилей сайта на виджет и наоборот.
3.  **Асинхронный мессенджинг:** Взаимодействие `content-script.js` <-> `background.js` (service worker) должно использовать строгую типизацию сообщений: `{ type: 'ACTION_NAME', payload: { ... } }`.

### Безопасность и Тарифы:
1.  **Прямые запросы запрещены:** Расширение ЗАПРЕЩЕНО настраивать на прямой запрос к `https://generativelanguage.googleapis.com`. Все запросы идут через ваш прокси-сервер.
2.  **Контекст тарифов** (подробности — см. `SPEC.md` → Контракты интерфейсов):
    * `Free`: Модель `gemini-2.5-flash`, лимит контекста 4k токенов, чтение только выделенного текста.
    * `Pro`: Модель `gemini-1.5-pro`, лимит контекста 32k токенов, чтение всего DOM-дерева.
    * `Ultra`: Модель `gemini-1.5-pro` (или актуальная flagship), лимит контекста до 100k+ токенов, расширенный контекст.

## 4. Инструкции по Взаимодействию (Интерфейс передачи задач)

Каждая фаза разработки (описанная в `SPEC.md`) должна выполняться изолированно. Приступая к задаче, агент обязан:
1.  **Прочитать текущую фазу из SPEC.md.** Не заглядывать в будущие фазы для оптимизации, если это нарушает атомарность текущей задачи.
2.  **Использовать только указанные интерфейсы.** Все типы сообщений, структуры payload и имена функций зафиксированы в секции **Контракты интерфейсов** SPEC.md. Запрещается создавать собственные типы сообщений.
3.  **Генерировать production-ready код.** Без плейсхолдеров вида `// TODO: реализовать позже`. Если метод указан — он должен быть написан полностью.
4.  **Логировать изменения:** По итогам выполнения фазы обновлять статус в `SPEC.md` (Метка: `[ВЫПОЛНЕНО]`).

> **Ссылка на спецификацию:** [`SPEC.md`](./SPEC.md) — содержит описание всех фаз, acceptance criteria и acceptance tests.

## 5. Git-Workflow (Обязательные правила)

Каждый агент **обязан** вести историю изменений через Git. Это обеспечивает прослеживаемость, возможность отката и прозрачность работы.

### Правила коммитов:
1.  **Один промпт = один коммит.** После завершения каждой фазы (или каждого крупного изменения по запросу пользователя) агент создаёт коммит.
2.  **Информативные сообщения.** Формат коммита: `feat(phase-X): краткое описание` (например, `feat(phase-1): add manifest, proxy server, and icons`).
3.  **Не коммитить секреты.** Файлы `.env`, `node_modules/`, и любые файлы из `.gitignore` НИКОГДА не попадают в индекс.
4.  **Проверять перед коммитом.** Перед `git add` проверять `git status` и `git diff`, чтобы убедиться в корректности добавляемых файлов.

### Порядок действий:
```
git add -A
git commit -m "feat(phase-X): описание изменений"
```
