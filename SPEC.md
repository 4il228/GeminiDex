# SPEC.md: Спецификация и пофазная декомпозиция разработки

Настоящий документ содержит атомарную структуру разработки расширения. Каждая фаза является изолированным техническим заданием (ТЗ). Исполнитель (агент) должен выполнять фазу, имея полное понимание требований и не допуская конфликтов с работой других агентов.

---

## Общие правила выполнения фаз

> **Каждая фаза — это отдельная задача.** Агент выполняет свою фазу самостоятельно. Каждая фаза содержит исчерпывающий список файлов, интерфейсов контрактов и ожидаемых результатов — агент НЕ ДОЛЖЕН изучать будущие фазы для оптимизации.

### Правило атомарности
1. **Читай только свою фазу.** Не изучай будущие фазы для «оптимизации» — это нарушает изоляцию задач.
2. **Создавай файлы по списку.** Каждая фаза содержит исчерпывающий список файлов для создания/модификации. Не создавай дополнительных файлов, не указанных в фазе.
3. **Не модифицируй файлы других фаз.** Если файл уже существует и относится к другой фазе — НЕ ИЗМЕНЯЙ его. Исключение: файлы, явно указанные в шагах текущей фазы как «Файлы для модификации».
4. **Ссылки на предыдущие фазы допустимы** в секции «Файлы, которые НЕЛЬЗЯ изменять» — это указывает агенту границы его ответственности, а не требует изучения чужого кода.

### Правило интерфейсов между фазами
Каждая фаза, которая Produces (производит) данные для других фаз, должна описывать **контракт интерфейса**:
*   **Типы сообщений** (для мессенджинга между content-script ↔ background).
*   **Структура данных** (JSON-схема для запросов/ответов).
*   **Имена функций** (которые будут вызываться из других фаз).

Это гарантирует, что агенты, работающие параллельно, не вступят в конфликт.

### Правило предотвращения ошибок
1. **Проверяй зависимости перед записью.** Перед созданием файла убедись, что родительская директория существует. Создавай её при необходимости.
2. **Используй полные пути.** Указывай относительные пути от корня проекта (например: `src/content/content-script.js`).
3. **Не используй `eval()` и динамические импорты.** Только статические ES-модули (ограничение Manifest V3).
4. **Изолируй стили.** Весь CSS для UI агента — только внутри Shadow DOM. Никаких глобальных стилей.
5. **Инжект файлов в Shadow DOM.** В Manifest V3 content script НЕ МОЖЕТ загружать локальные файлы через `<script src="...">`. Файлы `widget.js` и `styles.css` должны читаться как текст и инжектироваться через `textContent` / `innerHTML`.

### Правило верификации
По завершении фазы агент обязан:
1. Перечислить все созданные/изменённые файлы.
2. Указать, что каждый файл содержит (краткое описание назначения).
3. Отметить статус фазы: `[ВЫПОЛНЕНО]` в SPEC.md напротив номера фазы.

## Архитектурная схема потока данных

```
[User UI (Shadow DOM)] <-> [Content Script] <-> [Background Service Worker] <-> [Backend Proxy] <-> [Gemini API]
```

### Контракты интерфейсов (межфазные соглашения)

Для предотвращения конфликтов между агентами все интерфейсы фиксируются здесь. Агенты **обязаны** использовать только эти типы, структуры и имена.

#### Сообщения (Content Script ↔ Background Service Worker)

| Тип сообщения | Направление | Payload | Описание |
|---|---|---|---|
| `SET_AUTH_TOKEN` | CS → BG | `{ token: string }` | Сохранить JWT-токен |
| `GET_USER_TIER` | CS → BG | `{}` | Запросить текущий тариф |
| `GET_USER_TIER` (ответ) | BG → CS | `{ tier: 'free' \| 'pro' \| 'ultra' }` | Ответ с тарифом |
| `SEND_PROMPT` | CS → BG | `{ message: string, context?: string }` | Отправить запрос к ИИ |
| `STREAM_CHUNK` | BG → CS | `{ text: string }` | Чанк SSE-потока |
| `STREAM_END` | BG → CS | `{}` | Поток завершён |
| `STREAM_ERROR` | BG → CS | `{ error: string }` | Ошибка потока |

#### Тарифные конфигурации (Backend)

```javascript
const TIER_CONFIG = {
  free:  { model: 'gemini-2.5-flash', maxTokens: 4000,  contextLimit: '4k' },
  pro:   { model: 'gemini-1.5-pro',   maxTokens: 32000, contextLimit: '32k' },
  ultra: { model: 'gemini-1.5-pro',   maxTokens: 100000, contextLimit: '100k+' }
};
```

#### JWT Payload (структура токена)

```json
{
  "userId": "string",
  "tier": "free" | "pro" | "ultra",
  "iat": "number",
  "exp": "number"
}
```

## Фаза 1: Инфраструктура, Манифест и Прокси-Сервер (Запуск ядра)

**Цель:** Создать каркас расширения под Manifest V3 и защищенный Backend-слой для работы с тарифами.

**Роль исполнителя:** Архитектор + Бэкенд-инженер.

### Исходные требования:
*   Поддержка Яндекс Браузера (Chromium MV3).
*   API-ключ Gemini жестко изолирован на бэкенде (в `.env` файле, не в коде).
*   Бэкенд-прокси работает отдельно от расширения (отдельный процесс Node.js).
*   Бэкенд-сервер слушает адрес `http://localhost:3000` (настраивается через `.env`).

### Файлы для создания:
| Файл | Назначение |
|------|------------|
| `manifest.json` | Манифест расширения (MV3) |
| `.env.example` | Шаблон переменных окружения (без секретов) |
| `server/package.json` | Зависимости бэкенда |
| `server/index.js` | Точка входа Express-сервера |
| `server/middleware/auth.js` | Middleware для JWT-валидации |
| `server/routes/chat.js` | Эндпоинт `/api/v1/chat` |
| `icons/icon16.png` | Иконка 16x16 (placeholder PNG) |
| `icons/icon48.png` | Иконка 48x48 (placeholder PNG) |
| `icons/icon128.png` | Иконка 128x128 (placeholder PNG) |

### Шаги реализации:
1. **Манифест расширения:**
   * Создать `manifest.json` со следующим содержимым:
     * `manifest_version`: `3`
     * `name`: `"Gemini Agent"`
     * `version`: `"1.0.0"`
     * `permissions`: `["storage", "activeTab", "scripting"]`
     * `background.service_worker`: `"src/background/background.js"`
     * `content_scripts`: `[{ "matches": ["http://*/*", "https://*/*"], "js": ["src/content/content-script.js"], "run_at": "document_end" }]`
     * `action.default_popup`: `"src/ui/widget.html"` — **НЕ используется как popup**; определено для отображения иконки в панели расширения. Основной UI инжектируется content-script'ом через Shadow DOM.
     * `icons`: `{ "16": "icons/icon16.png", "48": "icons/icon48.png", "128": "icons/icon128.png" }`

2. **Переменные окружения:**
   * Создать `.env.example`:
     ```
     GEMINI_API_KEY=your_api_key_here
     JWT_SECRET=your_jwt_secret_here
     PORT=3000
     ```
   * Создать `.env` с реальными значениями (не коммитить!).
   * Добавить `.env` в `.gitignore`.

3. **Бэкенд-Прокси (Node.js/Express):**
   * Инициализировать `server/package.json` с зависимостями: `express`, `cors`, `dotenv`, `@google/genai`, `jsonwebtoken`.
   * Создать `server/index.js`:
     * Express-сервер на порту из `.env` (по умолчанию `3000`).
     * Подключить CORS для разрешения запросов из расширения.
     * Подключить JSON-парсер.
     * Подключить роутер `/api/v1`.
   * Создать `server/middleware/auth.js`:
     * Извлечь JWT из заголовка `Authorization: Bearer <token>`.
     * Декодировать токен (без верификации — верификация на стороне авторизующего сервиса).
     * Извлечь `userId` и `tier` из payload.
     * Прикрепить к `req.user`. При ошибке — ответ 401.
   * Создать `server/routes/chat.js`:
     * `POST /api/v1/chat` — принимает `{ message: string, context?: string }`.
     * Применяет middleware `auth`.
     * Маппит тариф из `req.user.tier` на конфигурацию модели (см. **Контракты интерфейсов** → Тарифные конфигурации).
     * Инициализирует `@google/genai` с API-ключом из переменных окружения.
     * Отправляет запрос к Gemini с включенным стримингом (`stream: true`).
     * Пробрасывает SSE-поток клиенту через `res.setHeader('Content-Type', 'text/event-stream')`.

4. **Иконки:** Создать заглушки (placeholder) для иконок 16x16, 48x48, 128x128 (любой валидный PNG, можно одноцветный).

### Ожидаемый результат:
* `manifest.json` валиден и расширение устанавливается в Яндекс Браузер в режиме разработчика.
* Бэкенд-прокси запускается через `node server/index.js`, эндпоинт `POST /api/v1/chat` отвечает SSE-потоком при передаче валидного JWT.
* API-ключ Gemini НЕ передаётся клиенту, НЕ хранится в коде расширения.

### Verification Checklist:
- [ ] `manifest.json` существует и содержит все указанные поля
- [ ] `.env.example` содержит все переменные окружения
- [ ] `server/package.json` содержит все указанные зависимости
- [ ] `server/index.js` экспортирует Express-приложение и слушает порт
- [ ] `server/middleware/auth.js` экспортирует функцию middleware
- [ ] `server/routes/chat.js` экспортирует Express-роутер
- [ ] Иконки созданы (3 файла PNG)

## Фаза 2: Изолированный UI и Внедрение на Страницу (Content Layer)

**Цель:** Разработать интерфейс ИИ-агента, который внедряется на любую вкладку и не зависит от стилей текущего сайта.

**Роль исполнителя:** Фронтенд-инженер.

### Исходные требования:
*   Полная изоляция через Shadow DOM.
*   Отсутствие утечек CSS (стили сайта не должны ломать виджет).
*   Файлы UI создаются впервые (не модифицируют существующие).
*   Content script ссылается на `manifest.json` (создан в Фазе 1) — агент должен проверить его наличие, но НЕ ИЗМЕНЯТЬ.

### Интерфейсы для использования:
*   Из Фазы 1: `manifest.json` содержит `content_scripts` с путём к `src/content/content-script.js`.
*   Для мессенджинга с Background Worker используются типы сообщений из **Контрактов интерфейсов** (см. общую секцию): `SEND_PROMPT`, `STREAM_CHUNK`, `STREAM_END`, `STREAM_ERROR`.
*   Структура payload для `SEND_PROMPT`: `{ message: string, context?: string }`.

### Файлы для создания:
| Файл | Назначение |
|------|------------|
| `src/content/content-script.js` | Скрипт внедрения и управления виджетом |
| `src/content/styles.css` | Стили виджета (изолированные через Shadow DOM) |
| `src/ui/widget.html` | HTML-шаблон виджета (внедряется в Shadow DOM как текст) |
| `src/ui/widget.js` | Логика UI: ввод, отправка, отображение ответов |

### Файлы, которые НЕЛЬЗЯ изменять:
* `manifest.json` (создан в Фазе 1)
* `server/*` (создано в Фазе 1)

### Шаги реализации:
1. **Скрипт внедрения (`content-script.js`):**
   * При загрузке страницы (событие `DOMContentLoaded` или `run_at: document_end`):
     * Создать `<div id="gemini-agent-root">`.
     * Прикрепить к нему Shadow DOM (`attachShadow({ mode: 'open' })`).
     * **Прочитать содержимое** `widget.html`, `widget.js` и `styles.css` как текст (используя `fetch(chrome.runtime.getURL(...))` или инлайн-строки).
     * Внедрить CSS-текст в `<style>` тег внутри Shadow DOM.
     * Внедрить HTML-текст как innerHTML Shadow Root.
     * Внедрить JS-текст через `document.createElement('script')` → `script.textContent = jsCode` →追加 в Shadow DOM.
   * Реализовать обработчик сообщений от Background Worker (типы сообщений — см. **Контракты интерфейсов**):
     * `STREAM_CHUNK` — добавлять текст в окно чата.
     * `STREAM_END` — завершить отображение ответа.
     * `STREAM_ERROR` — показать сообщение об ошибке.

2. **Стили виджета (`styles.css`):**
   * Все стили ТОЛЬКО через селекторы внутри Shadow DOM (`:host`, `.chat-container`, etc.). Никаких глобальных селекторов.
   * Основные компоненты:
     * Фиксированная позиция: `position: fixed; right: 0; top: 0; height: 100vh; width: 350px; z-index: 2147483647;`
     * Фон: `background: #ffffff; box-shadow: -2px 0 8px rgba(0,0,0,0.1);`
     * Скроллируемая область чата: `overflow-y: auto; flex: 1;`
     * Поле ввода: фиксированное внизу панели.
     * Кнопка-триггер: для сворачивания/разворачивания панели (позиция: `fixed; right: 10px; top: 50%; z-index: 2147483647;`).

3. **HTML-шаблон (`widget.html`):**
   * Минимальная структура:
     * `<div class="chat-container">` — контейнер чата.
     * `<div class="messages">` — область сообщений.
     * `<div class="input-area">` — область ввода с `<textarea>` и `<button>`.
   * Этот файл инжектируется как **текст** в Shadow DOM, а не загружается как отдельная страница.

4. **Логика UI (`widget.js`):**
   * Перехватить отправку сообщения (клик по кнопке или `Enter`).
   * Считать текст из `<textarea>`.
   * Отправить через `chrome.runtime.sendMessage({ type: 'SEND_PROMPT', payload: { message: text } })`.
   * Реализовать функцию `appendMessage(text, role)` для добавления сообщений в DOM.
   * Реализовать функцию `appendStreamChunk(text)` для посимвольного добавления ответа ИИ.
   * Реализовать автоскролл: `messages.scrollTop = messages.scrollHeight`.

### Ожидаемый результат:
* На любом открытом сайте в Яндекс Браузере появляется боковая панель (350px справа).
* Стили сайта (глобальные сбросы `margin`, `font-family`, `* { box-sizing: ... }`) НЕ влияют на виджет.
* Пользователь может ввести текст и отправить сообщение (оно уйдёт в Background Worker).
* Панель сворачивается/разворачивается по кнопке.

### Verification Checklist:
- [ ] `content-script.js` создаёт Shadow DOM и внедряет UI
- [ ] `styles.css` не содержит глобальных селекторов (все стили — через `:host` или контейнер Shadow DOM)
- [ ] `widget.html` содержит minimal markup для чата
- [ ] `widget.js` корректно отправляет сообщения через `chrome.runtime.sendMessage`
- [ ] Автоскролл работает
- [ ] Виджет не ломает верстку тестовой страницы
- [ ] Файлы JS/CSS инжектируются как текст, не через `<script src>`

## Фаза 3: Логика Background Worker и Авторизация (State Layer)

**Цель:** Связать UI расширения, локальное хранилище и бэкенд, обеспечив бесшовную авторизацию и сессии.

**Роль исполнителя:** Архитектор + Фронтенд-инженер.

### Исходные требования:
*   `background.js` работает как единая точка отправки сетевых запросов.
*   Токен авторизации хранится в `chrome.storage.local`.
*   Content Script (Фаза 2) уже отправляет сообщения типа `SEND_PROMPT`.
*   Бэкенд-сервер доступен по адресу `http://localhost:3000` (настраивается через `.env`).

### Интерфейсы для использования:
*   Типы сообщений и их payload — см. **Контракты интерфейсов** (общая секция).
*   Функция `getPageContext()` из Фазы 2 **ещё не существует** — Phase 3 работает без неё.

### Файлы для создания:
| Файл | Назначение |
|------|------------|
| `src/background/background.js` | Service Worker: обработка сообщений, запросы к бэкенду |

### Файлы для модификации:
| Файл | Что изменить |
|------|--------------|
| `manifest.json` | Добавить `host_permissions` для бэкенд-сервера |

### Файлы, которые НЕЛЬЗЯ изменять:
* `src/content/*` (создано в Фазе 2)
* `src/ui/*` (создано в Фазе 2)
* `server/*` (создано в Фазе 1)

### Шаги реализации:
1. **Background Worker (`background.js`):**
   * Слушатель установки/обновления расширения:
     * `chrome.runtime.onInstalled.addListener` — инициализировать `chrome.storage.local` с дефолтными значениями (`{ authToken: null, userTier: 'free' }`).
   * Обработчик сообщений `chrome.runtime.onMessage.addListener`:
     * Тип `SET_AUTH_TOKEN`: сохранить `payload.token` в `chrome.storage.local` под ключом `authToken`.
     * Тип `GET_USER_TIER`: декодировать JWT из `chrome.storage.local` (без верификации), извлечь поле `tier`, вернуть `{ tier: decoded.tier }`. Если токена нет — вернуть `{ tier: 'free' }`.
     * Тип `SEND_PROMPT`:
       1. Считать `authToken` из `chrome.storage.local`.
       2. Если токена нет — вернуть ошибку `{ type: 'STREAM_ERROR', payload: { error: 'Unauthorized' } }`.
       3. Сформировать `fetch` к бэкенду: `POST http://localhost:3000/api/v1/chat` с заголовком `Authorization: Bearer <token>` и телом `{ message: payload.message, context: payload.context }`.
       4. Обработать SSE-ответ: читать `ReadableStream`, для каждого чанка отправлять `{ type: 'STREAM_CHUNK', payload: { text: chunk } }` в Content Script.
       5. По завершении потока отправить `{ type: 'STREAM_END' }`.

2. **Манифест (обновление):**
   * Добавить в `manifest.json` поле `host_permissions`: `["http://localhost:3000/*"]` (адрес бэкенда из `.env`).

### Ожидаемый результат:
* Пользователь может «авторизоваться»: токен сохраняется в `chrome.storage.local`.
* Сообщение `SEND_PROMPT` из Content Script проходит через Background Worker на бэкенд и возвращается SSE-потоком.
* Если токен отсутствует — возвращается ошибка `Unauthorized`.
* Background Worker НЕ хранит API-ключ Gemini, НЕ делает прямых запросов к Google.

### Verification Checklist:
- [ ] `background.js` обрабатывает все три типа сообщений (`SET_AUTH_TOKEN`, `GET_USER_TIER`, `SEND_PROMPT`)
- [ ] `chrome.storage.local` используется для хранения токена
- [ ] Запросы к бэкенду идут через `fetch` с заголовком `Authorization`
- [ ] SSE-поток корректно пробрасывается в Content Script
- [ ] `manifest.json` содержит `host_permissions`
- [ ] `GET_USER_TIER` возвращает `{ tier: 'free' }` по умолчанию, если токен отсутствует

## Фаза 4: Чтение контекста страницы и Стриминг (Agent Layer)

**Цель:** Оснастить агента функциями чтения содержимого вкладки в зависимости от тарифа и финализировать интерфейс.

**Роль исполнителя:** Фронтенд-инженер + Бэкенд-инженер.

### Исходные требования:
*   Для тарифа `free` — чтение только выделенного текста.
*   Для тарифа `pro` — чтение всего DOM-дерева (текстового содержимого) активной вкладки.
*   Для тарифа `ultra` — чтение всей страницы + системные промпты (расширенный контекст).
*   UI виджет (Фаза 2) и Background Worker (Фаза 3) уже работают.
*   Для определения тарифа пользователя использовать `GET_USER_TIER` (см. **Контракты интерфейсов**).

### Интерфейсы для использования:
*   Тип `GET_USER_TIER` → ответ `{ tier: 'free' | 'pro' | 'ultra' }` (из Фазы 3).
*   Тип `SEND_PROMPT` с payload `{ message: string, context?: string }` (в Фазе 4 поле `context` заполняется).
*   Типы `STREAM_CHUNK`, `STREAM_END`, `STREAM_ERROR` — для отображения ответа (уже обрабатываются в Фазе 2).

### Файлы для модификации:
| Файл | Что изменить |
|------|--------------|
| `src/content/content-script.js` | Добавить сбор контекста страницы |
| `src/ui/widget.js` | Добавить переключатель «Анализ страницы», передачу контекста, получение тарифа |
| `server/routes/chat.js` | Обработка контекста (если передан) |
| `server/middleware/auth.js` | Учесть расширенные права для `pro`/`ultra`-тарифа (опционально) |

### Файлы, которые НЕЛЬЗЯ изменять:
* `manifest.json` (изменялся в Фазе 3)
* `src/background/background.js` (создан в Фазе 3)

### Шаги реализации:
1. **Сбор контекста (`content-script.js`):**
   * Реализовать функцию `getPageContext(tier)`:
     * Если `tier === 'free'`: вернуть выделенный текст `window.getSelection().toString().trim()` (или пустую строку).
     * Если `tier === 'pro'`: вернуть `document.body.innerText`.
     * Если `tier === 'ultra'`: вернуть `document.body.innerText` (в будущем — расширенный контекст с системными промптами).
   * При отправке сообщения `SEND_PROMPT`:
     1. Сначала запросить тариф: `chrome.runtime.sendMessage({ type: 'GET_USER_TIER' }, (response) => { ... })`.
     2. Получив `{ tier }`, вызвать `getPageContext(tier)`.
     3. Добавить к payload поле `context`: `{ message: text, context: pageContext }`.

2. **UI для контекста (`widget.js`):**
   * При инициализации запросить тариф: `chrome.runtime.sendMessage({ type: 'GET_USER_TIER' })` и вызвать `setTierUI(tier)`.
   * Добавить checkbox/переключатель: «Анализировать страницу» (виден только для `pro` и `ultra` тарифов).
   * Реализовать функцию `setTierUI(tier)`:
     * `free`: скрыть checkbox «Анализ страницы».
     * `pro`/`ultra`: показать checkbox.
   * При отправке сообщения: если переключатель активен (или тариф `pro`/`ultra`), прикреплять контекст.

3. **Бэкенд (`server/routes/chat.js`):**
   * Если в запросе есть поле `context` — добавлять его к промпту перед отправкой в Gemini:
     ```
     Контекст страницы:
     {context}

     Вопрос пользователя: {message}
     ```
   * Если `context` пуст — отправлять только `message`.

4. **Рендеринг Markdown (`widget.js`):**
   * Реализовать простой парсер Markdown для ответов ИИ:
     * `**жирный**` → `<strong>`
     * `` `код` `` → `<code>`
     * ` ```код``` ` → `<pre><code>`
     * `\n` → `<br>`
   * Или использовать lightweight-библиотеку (например, `marked` minified) — инжектировать в Shadow DOM.

5. **Стриминг в UI (`widget.js`):**
   * При получении `STREAM_CHUNK`: добавлять текст в текущий элемент ответа ИИ.
   * При `STREAM_END`: завершить элемент, включить кнопку отправки.
   * При `STREAM_ERROR`: показать сообщение об ошибке, включить кнопку отправки.

### Ожидаемый результат:
* Полностью рабочий ИИ-агент в боковой панели Яндекс Браузера.
* Агент отвечает по контексту открытой статьи/сайта (для `pro`/`ultra`-тарифа).
* Лимиты токенов соответствуют тарифу из JWT (4k для `free`, 32k для `pro`, 100k+ для `ultra`).
* Ответы ИИ отображаются с форматированием Markdown (жирный, код, списки).
* Панель плавно скроллится по мере поступления ответа.

### Verification Checklist:
- [ ] `getPageContext(tier)` возвращает корректный контекст в зависимости от тарифа
- [ ] Контекст передаётся на бэкенд в поле `context`
- [ ] Бэкенд приклеивает контекст к промпту перед отправкой в Gemini
- [ ] Markdown рендерится корректно (жирный, код, списки)
- [ ] Автоскролл работает при стриминге
- [ ] UI корректно скрывает/показывает элементы для разных тарифов
- [ ] `GET_USER_TIER` вызывается при инициализации виджета
- [ ] Расширение протестировано на 2+ разных сайтах без багов верстки
